# CLAUDE.md

법조문 형식의 사내 규정집을 대상으로 하는 온프레미스 RAG 시스템 구축 프로젝트입니다.
상세한 조사 배경은 `docs/RAG.md` 를 참조하세요.

---

## 1. 환경 제약 (반드시 준수)

- **온프레미스 + 로컬 LLM**: 외부 임베딩/LLM API 호출 금지 (OpenAI, Cohere, Anthropic 등 모든 외부 API 불가). 로컬 LLM 모델 사용.
- **GPU 자원 제한**: 모델 크기·배치 사이즈 선택 시 항상 메모리를 고려할 것.
- **문서 규모**: 약 100쪽 규정집. 
- **문서 구조**: 편-장-절-조-항-호 법조문 체계. '조' 단위 완결성을 기준으로 설계됨.

---

## 2. 확정 설계 (임의로 다른 방식으로 대체하지 말 것)

- **청킹**: 구조 기반(조 단위) + Parent-Child. 항 단위로 검색하고, LLM에는 조 전체를 전달.
  - 표는 위에 요약 텍스트를 붙여 함께 인덱싱.
- **임베딩**: 한국어 로컬 모델 (Snowflake/snowflake-arctic-embed-l-v2.0). 최종 모델은 실험으로 확정.
- **검색**: Hybrid (BM25 + 벡터), RRF로 결합. + 로컬 Re-ranker (cross-encoder).
- **메타데이터**: 편-장-절-조 경로를 청크마다 저장. 메타데이터 필터링으로 범위 선제 축소.
- **구조 수준**: Advanced부터 시작 (검색 전후 보정 단계 포함). 이후 필요 시 고도화.

---

## 3. 임베딩 단계

- 임베딩이 일어나는 두 시점:
  1. 인덱싱 시점(1회): 청킹한 규정집 조각들을 벡터로 변환해 벡터 DB에 저장.
  2. 검색 시점(매 질문마다): 사용자 질문을 벡터로 변환 → 가장 가까운 조각 검색.
- 질문과 문서는 반드시 동일한 임베딩 모델로 변환할 것 (벡터 공간이 같아야 비교 가능).
- 질문 임베딩은 매 요청마다 발생하는 병목 구간. 모델 크기 선택 시 이 지연을 고려.

---

## 4. 실험으로 결정할 변수 (하드코딩 금지, config로 분리)

아래 값들은 실험 결과로 정해지므로 코드에 직접 박지 말고 설정 파일로 분리하여 변경·재현이 가능하게 구성할 것.

- 임베딩 모델 (후보 2~3개 비교)
- RRF의 sparse / dense 가중치 비율
- top-k 값
- Re-ranker 적용 여부 (정확도 향상분 vs 지연 트레이드오프)
- 청킹 방식: 조 단위 단일 청크 vs Parent-Child
- ANN 알고리즘 적용 여부 (속도 향상분 vs 정확도 트레이드오프)
---

## 5. 평가 체계

- **지표**: Hit Rate@k, MRR, 최종 답변 품질, RAGAS 지표.
- **평가셋**: 질문-정답 약 50개. 질문 유형을 일상어 질문 / 조항·용어를 직접 언급하는 질문으로 구분.
- **원칙**: 모든 실험은 동일 질문셋으로 비교. 결과는 표 형태로 기록.

### 진행 예정 실험
1. 검색 방식 비교: Sparse 단독 / Dense 단독 / Hybrid (RRF 비율 튜닝 포함).
2. 청킹 비교: 조 단위 단일 청크 / Parent-Child.
3. Re-ranker 유무 비교: 정확도 향상분과 지연 증가를 함께 측정.
4. 임베딩 모델 비교: 정확도가 충분히 나오는 가장 작은 모델 탐색.

---

## 6. 기술 스택 & 코딩 규칙

- **언어**: Python
- **벡터 DB**: Qdrant, 로컬 파일 임베디드 모드(서버 불필요, `QdrantClient(path=...)`) — 로컬·소규모, ANN 불필요.
- **설정 분리**: 실험 변수는 모두 config 파일로 관리.
- **재현성**: 랜덤 시드 고정.
- **문서화**: 주석·문서는 한국어로 작성.

---

## 7. 작업 시 주의 (Negative 지시)

- 외부 API를 쓰는 코드를 제안하거나 작성하지 말 것.
- GraphRAG, Hierarchical Indexing(RAPTOR 등)은 후순위. 지금 단계에서 구현하지 말 것.
- 실험으로 정할 값을 임의로 확정해 하드코딩하지 말 것.
- 진행 순서: 청킹 → 인덱싱 → 검색 → 평가. 이 순서를 따를 것.
