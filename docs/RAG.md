# 1. RAG

# RAG (검색 증강 기법)

> AI가 두꺼운 책에서 필요한 부분을 찾아 읽고 답하는 것
> 

## 프로젝트 제약

- **온프레미스 + 로컬 LLM**: 외부 임베딩 API 사용 불가, GPU 자원 제한
- **소규모 문서**: 100쪽 규정집 → 조각 수백~수천 개 수준 (ANN 최적화 불필요, 완전 탐색으로도 충분)
- **고도로 구조화된 문서**: 편-장-절-조-항-호 법조문 체계, '조' 단위 완결성 설계 → 구조 기반 청킹에 최적

## 필수 적용 사항

1. 청킹 / 인덱싱 전략

```markdown
# 문서를 어떻게 자르고(청킹) 어떻게 정리할지(인덱싱)
# 원하는 내용만 빠르고 정확하게 읽기 위함
# 어떻게 자료를 청킹할꺼냐?

# 청킹 전략
1. 고정 크기 청킹: 글자 수나 토큰 수를 정해놓고 자르는 방법.
	a. 장점: 구현이 쉽고 빠름.
	b. 단점: 의미를 무시하고 자름.
		ex) '사자는 초원에 살며 사자의 역할은'에서 끊김. 역할의 의미를 알 수 없음.
	c. 보안방법: 오버랩(overlap): 500토큰 씩 자르되 앞 조각의 마지막 50토큰을 다음 조각 앞에 넣음.

2. 재귀적 청킹(Recursive chunking): 큰 단위(문단)부터 작은 단위(키워드)로 내려가며 충분히 조각이 작아질 때까지(미리 지정) 자르기를 반복.
	a. 장점:
		1. 의미 단위를 최대한 지키면서 크기 제한도 맞춘다.
		2. 의미 기반 청킹과는 달리 임베딩 계산(문장을 벡터로 바꿔 의미 비교)이 필요없어, 비용이 거의 안들고 처리속도가 빠르다. 
		3. 범용성이 크다. 문서 종류를 안 가리고 무난한 기본값.
	b. 단점:
		1. 진짜 의미는 이해하지 못한다. 항상 구분선이 의미 경계와 일치하지 않을 뿐더러 구분선(문단, 문장)은 지키지만 한문단에서 주제가 둘로 나뉘어 모른다.
		2. 구분선이 약한 문서에서 효과가 좋지 않다. 표나 코드처럼 일반 문장 구조가 아닌 데이터에 적용하기 어렵다.
		
3. 의미 기반 청킹(Sementic chunking): 문자를 임베딩으로 변환한 다음, 이웃한 문장끼리 의미가 얼마나 비슷한지 비교하여 자른다.
	a. 장점:
		1. 임베딩으로 문장간 의미를 비교하여 한 조각으로 묶기 때문에 검색이 정확해진다.
		2. 재귀적 청킹보다 느리고 돈이 더 많이 든다.
	b. 단점:
		1. 유사도가 얼마나 떨어져야 자를지 임계값(threshold)을 잘못 잡으면 너무 쪼개지거나, 안 잘리기도 함.
		
4. 문서 구조 기반 청킹: MD파일의 제목(헤더), HTML 태그, PDF의 섹션 단위로 자르는 방법. 구조가 잘 잡힌 문서에 용이함.
	a. 장점:
		1. 사람이 의도한 의미 경계를 그대로 가져올 수 있다.
		2. 빠르고 저렴하다. 임베딩 계산이 필요없고 헤더나 태그를 찾아 자르는 규칙 기반이라 비용이 거의 안 든다.
		3. 맥락 정보(메타데이터)를 함께 보존하기 좋다. 해당 조각이 어떤 문서에 속한 내용인지 알 수 있어 정확도와 신뢰도가 올라간다.
	b. 단점:
		1. 구조가 엉성한 문서에는 못 쓴다. 구조 의존도가 매우 높다.
		2. 조각 크기가 각기 다른다. 너무 긴 섹션은 여러 내용이 들어가 검색이 흐려진다.
		-> 너무 긴 섹션은 재귀적 청킹으로 한번 더 자르는 방법으로 타파.

[임베딩]: 글(단어, 문장)을 벡터로 바꾸는 것, 숫자로 바꾸면 의미가 가까운지 거리로 판단할 수 있음.
```

```markdown
# 인덱싱 전략
# 청킹으로 조각으로 자른 것을 질문이 들어왔을 때 더 빠르게 찾아주기 위해 정리하는 것.
# 어떤 임베딩 모델을 쓸거냐? 어떤 ANN 알고리즘/인덱스 구조를 쓸거냐? 무엇을 기준으로 인덱싱 할꺼냐?

1단계: 조각을 벡터로 저장
	청킹한 조각들을 임베딩하여 벡터로 바꾼 후 벡터DB에 저장한다. (벡터DB: Pinecone, Weaviate, Chroma, FAISS)
2단계: 질문이 들어오면 가까운 걸 찾기
	사용자 질문 -> 질문을 임베딩하여 벡터로 바꾸기 -> 질문 벡터와 가장 가까운 조각 벡터 찾기
	보통 가장 가까운(의미가 비슷한) 상위 몇가지(top-k, 예를 들어 상위 5개)를 가져온다.
		- 문제: 조각이 100만 개면 100만 개를 비교(완전 탐색)하기 때문에 너무 느리다.
	ANN(Approximate Nearest Neighbor): 근사 최근접 이웃, 거의 가까운 걸 훨씬 빠르게 찾는법. 정확도는 떨어지나 속도를 얻음.
		- 예시: 서울에서 나랑 제일 가까운 카페를 찾을 때 전국 카페를 비교하지 않고 우리 동네에서 찾기.
		HNSW 기법 사용(미리 정리해두고 가까운 걸 빨리 찾는 구조) 
```

1. Parent-Child / Small-to-Big: 작은 청크로 검색하고 큰 맥락을 전달  
2. Contextual Retrieval: 청크에 문맥 정보를 추가해 검색 정확도 향상

1. 검색 방식

```markdown
# 무엇으로 문서에서 찾는지

1. Dense Retrieval(조각) - 의미(벡터)로 찾기
	: 질문과 조각을 벡터로 바꿔서 의미가 가까운 조각을 찾기
	
	장점: 동의어, 돌려말하기, 의역에 강함
	단점: 임베딩 비용이 듬. 단어 매칭에 약할 수 있음.
2. Sparse Retrieval(단어) - 단어(키워드)로 찾기
	: 질문에 들어있는 단어가 조각에도 들어있는지. 가장 옛날부터 쓰던 방식.
	가장 잘 알려진 방법으로는 BM25가 있다. 
	
	저장 구조는 역색인
	- 예시: 책 맨 뒤의 찾아보기 사사 -> 3쪽, 7쪽, 12쪽 단어가 나온 목록 
	
	장점: 정확한 단어 매칭이 중요한 경우 강하다.
	단점: 뜻을 모른다. 동의어, 돌려 말하면 잘 찾기 못한다.   
3. Hybird Search(조각 + 단어)
	: 질문이 들어오면 키워드 검색과 의미 검색을 같이 함. 각각의 결과 점수를 합쳐 최종 순위를 매김.
	이때 사용하는 방법은 RRF(Reciprocal Rank Fusion)이다.
	실무에서 가장 많이 쓰는 출발점
```

## 선택적 적용 사항

1. 기본 구조 수준

```markdown
# 정교함의 수준 (결과를 보고 선택)

Naive(기본형) -> Advanced(보정형) -> Modular(조립형)

Naive: 질문을 그대로 검색 -> 조각을 그대로 프롬프트에 넣음 -> AI 답한다.
중간에 보정이 없음. 질문이 애매하거나 검색 결과가 애매해도 그래도 AI한테 넘김.
[참고 자료] : 조각...
[질문] 회사에서 이거하려는데 ...

Advanced: Naive에 검색전후 보정 단계를 끼워넣음. 고도화 기법이 여기에 끼워짐.
검색전 -> 질문을 다듬음(쿼리 재작성, 쿼리 확장) 검색후(AI한테 물어볼 때) -> re-ranking, 조각들을 한번더 재순위화, 압축

Modualr: 가장 유연한 상태: 레고 블럭처럼 기능들을 붙일 수 있음.
예시: 검색이 필요 없는 질문은 검색을 skip, 검색을 여러번 하기, 웹 검색을 추가로 하기 등등
```

1. 고도화 기법

```markdown
# 품질이 부족할 때 추가

1. Re-ranking(검색 품질을 높이기)
	: 속도를 위해 ANN과 같이 대충 가까운 것을 먼저 가져온 후, 세밀하게 줄 세우기
	1차 검색보다 무겁지만 정확한 모델(cross-encoder 등)을 써서 순위를 다시 매김. 가성비가 좋아서 기본으로 깔고 감.

2. HyDE(질문을 다듬기, 쿼리 변환)
	: 질문에 대한 가상의 답을 AI가 먼저 지어내고 그 가상 답으로 검색.
3. multi-query(한 질문을 여러 각도로 쪼개 검색) 
	
4. CRAG(검색 결과를 AI가 검증보정)
	: 검색 품질을 점검해서 너무 나쁘면 웹 검색같은 수단으로 보완
	
5. GraphRAG(관계를 활용)
	: 지식 그래프(개체들이 어떻게 연결되어있는지를 표현한 그물망)로 정리해서 검색.
	여러 정보를 연결해야 답할 수 있는 복잡한 질문에 좋다.
	
6. Agentic(AI가 스스로 판단하고 행동)
```

> Hit Rate: 적중률 → 정답이 상위 k개 안에 들어왔냐 (예시) Hit Rate@5 = 90% → 상위 5개안에 들어올 확률이 90%
 MRR: 평균 역순위 → 정답이 몇번째에 있었냐 (예시) 1.0(만점) 정답이 1번째 0.5 1/2 정답이 두번째, 0.33 1/3 정답이 세번째, …
zero-shot : 평가에 쓰인 데이터를 미리 학습하지 않은 정도
B: 10억, M: 100만

## 실험 후보

측정지표: Hit Rate, MRR, 최종 답변 품질

필요: 평가용 질문셋(약 50개) - “질문정답“ 세트

실험 이유: 일반론으로 고른 선택이 우리 문서에 맞는지 숫자로 확인.

- 실험 1: (검색 방식 비교) Sparse 단독 / Dense 단독 / Hybrid 세 가지를 같은 질문셋으로 돌려 어떤 질문 유형에서 누가 강한지 실험.
    - Hit Rate, MRR 비교. Hybrid가 단독들 보다 얼마나 높은지
        - Hybrid RRF(sparse + dense 섞는 비율)는 몇으로 하는게 최적인지
    - 질문 유형별(일상어, 조항|용어를 말하는 질문)
- 실험 2: (청킹 비교) 조 단위 단일 청크 / Parent-Child 두 가지를 비교.
    - Hit Rate, MRR 비교. 최종답변 품질 비교
- 실험 3: (Re-ranker 유무 비교) 정확도 향상분과 늘어난 응답시간을 함께 측정해 트레이드오프를 수치로 확인.
    - MRR 위주 비교.
    - Re-ranker를 거치면서 질문당 몇초가 더 걸리는지 교환비. MRR +0.15 / 지연 +0.3초 같은 숫자로 구체화
- 실험 4: (임베딩 모델 비교) 후보 한국어 모델 2~3개를 같은 조건에서 비교.
    - 검색 정확도가 충분히 나오는 가장 작은 모델은 무엇인지 → 로컬 환경 구축 이득

## RAG 방식 비교표 (트레이드오프)

| 선택 축 | 후보 | 장점 | 단점 (트레이드오프) | 적합도 |
| --- | --- | --- | --- | --- |
| 청킹 | 고정 크기 | 단순·빠름 | 조항/문장 중간 절단, 구조 무시 | 낮음 |
| 청킹 | 구조 기반 (조 단위) | 문서 설계와 일치, 의미 경계 정확, 저비용 | 긴 조항은 내부 다주제 | 높음 |
| 청킹 | Parent-Child | 검색 정밀 + 답변 맥락 동시 확보 | 구현 약간 복잡, 저장 중복 | 매우 높음 |
| 검색 | Sparse (BM25) | 조항번호·용어·수치에 강함, 가벼움 | 일상어↔규정용어 간극 못 메움 | 단독 부족 |
| 검색 | Dense (로컬 임베딩) | 의역·동의어 질문에 강함 | 정확한 코드·번호에 약함, GPU 필요 | 단독 부족 |
| 검색 | Hybrid (BM25+벡터) | 양쪽 강점 결합, 질문 유형 폭넓게 커버 | RRF 가중치 튜닝, 구현 복잡 | 매우 높음 |
| 임베딩 | 소형 로컬 모델 | 빠름, 자원 절약 | 한국어 미묘한 의미 구분 약함 | 중간 |
| 임베딩 | 대형 로컬 모델 | 한국어 정확도 높음, 장문 검색 강함 | GPU 메모리·시간 더 소모 | 높음 (소규모라 감당 가능) |

| 고도화 | GraphRAG | 조항 간 참조 추론 | 구축 복잡·고비용 | 낮음 (후순위) |

## 추천 조합

- **청킹**: 구조 기반(조 단위) + Parent-Child. 항 단위 검색, 조 전체를 LLM에 전달. 표는 위에 요약 텍스트를 붙여 함께 인덱싱.
- **인덱싱**: 한국어 특화 로컬 임베딩 모델(KURE 또는 BGE-M3 계열). 소규모라 ANN 최적화 불필요. 편-장-절-조 경로를 메타데이터로 저장.
- **검색**: Hybrid (BM25 + 벡터) + 로컬 Re-ranker.
- **구조**: Advanced부터 시작하여 추가 고도화.
- **추가 적용 권장**: 메타데이터 필터링 (장 단위 범위 축소).
